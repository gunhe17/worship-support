Worship Support (worship-support)
================================

1) 기본 설명
-----------
이 프로젝트는 예배/행사용 콘텐츠를 “템플릿(template) + 블록(block)” 조합으로 편집하고, 화면(슬라이드/포스터/게시물 등) 레이아웃에 배치(render)하여 출력/공유할 수 있는 웹서비스를 목표로 합니다.

- Project: 작업 단위(테넌트). 프로젝트마다 템플릿/블록이 분리됩니다.
- Template: 산출물 유형(예: poster, bulletin, presentation)별 레이아웃의 그릇입니다.
- Block: 재사용 가능한 콘텐츠 조각입니다(텍스트/이미지/일정/찬양/목록/광고/배경 등).
- Render: 특정 템플릿에 특정 블록을 어디에 어떻게 배치할지(좌표/크기/레이어 등)를 정의합니다.

블록은 “현재 버전(current) + 버전 히스토리(history)”를 함께 다루는 방식으로 설계합니다.
즉 block 테이블은 공통 메타데이터와 current_version_id를 가지고, 실제 콘텐츠는 block_txt, block_image 같은 타입별 테이블에 버전 row로 저장합니다.


2) 개발 도구(권장 기준)
----------------------
본 프로젝트는 Vercel + Supabase 기반의 서버리스 구성을 기본 전제로 합니다.

- 프론트엔드/서버 프레임워크: Next.js (App Router) + TypeScript
  - UI 라우팅/서버 액션/서버 렌더링(SSR)까지 하나의 코드베이스로 운영합니다.
- 배포/런타임: Vercel
  - 기본은 Node.js 런타임을 우선으로 사용하고, Edge 런타임은 꼭 필요할 때만 선택합니다.
- 인증/SSR 통합: Supabase Auth + @supabase/ssr
  - 기존 auth helper 계열(@supabase/auth-helpers-*) 대신 @supabase/ssr 흐름을 기준으로 통합합니다.
- 데이터베이스: Supabase Postgres
  - Row Level Security(RLS)와 정책(Policies)을 사용해 서버리스 환경에서도 안전한 데이터 접근을 보장합니다.
- 로컬 백엔드: Supabase CLI
  - 로컬에서 Postgres/Auth/Storage 등 Supabase 스택을 실행해 개발/테스트합니다.
  - Docker API 호환 런타임이 필요합니다(Docker Desktop 등).
- 서버리스 백엔드(역할 분리 권장):
  - 앱 내부 API/서버 액션: Next.js(Route Handlers/Server Actions) 중심
  - 외부 웹훅/서드파티 연동: Supabase Edge Functions(Deno/TypeScript) 중심
- 운영/진단 도구:
  - Vercel CLI: 프로젝트 연결, 로그 확인, 운영 작업 보조
  - Supabase Dashboard/CLI: DB 마이그레이션, Edge Functions 배포, 정책/로그 확인

옵션(기획에 따라 도입):
- AI 기능이 포함된다면 Vercel AI SDK를 검토(LLM 연동 보일러플레이트 감소)


3) 로컬 개발 흐름(예시)
----------------------
전제:
- Node.js LTS + 패키지 매니저(pnpm 권장)
- Docker API 호환 런타임
- Supabase CLI

흐름:
1) Supabase 로컬 스택 실행: `supabase start`
2) 환경변수 설정: Supabase URL/Anon key(로컬/개발/운영 분리)
3) 웹앱 실행: `next dev`
4) DB 변경은 마이그레이션으로 관리하고(DDL/정책 포함), 로컬에서 먼저 검증 후 반영합니다.


4) 운영 원칙(필수 체크)
----------------------
- 보안/패치: Next.js/React 및 의존성은 패치 릴리즈 라인을 지속적으로 최신 유지합니다.
- 권한/RLS: 서버에서 직접 DB에 접근하는 코드와, 클라이언트에서 Supabase를 통해 접근하는 코드를 혼용할 경우
  반드시 RLS/정책을 기준으로 “허용되는 데이터”를 정의합니다.
- 마이그레이션: 스키마/정책/트리거는 코드로 관리하고, PR 단위로 리뷰/적용합니다.
- 관측/장애 대응: Vercel 로그 + Supabase 로그를 기본 진단 루트로 잡고,
  장애 시나리오(인증, DB, 스토리지, 외부 웹훅)를 체크리스트화합니다.


5) 데이터 모델(초안 요약)
-----------------------
- project(id, name, created_at)
- template(project_id, name, type, created_at)
- block(project_id, type, current_version_id, created_at)
  - type 예: txt, image, datetime, song, song_list, advertisement, background
- block_{type} 테이블들: block_txt, block_image, block_datetime, block_song, block_song_list, block_advertisement, block_background
  - 공통: block_id(FK), title/description(선택), created_at, created_by
- 컴포지션:
  - block_song_list_item(song_list_block_id, song_block_id, sequence)
  - block_advertisement_item(advertisement_block_id, child_block_id, sequence)
- 렌더:
  - render(template_id, block_id, location, size)

주의:
- block.current_version_id는 타입별 테이블의 “특정 버전 row”를 가리키는 폴리모픽 관계이므로,
  무결성은 트리거/애플리케이션 레벨 검증으로 보강하는 것을 권장합니다.
