원하시는 것은:

- 특정 패턴을 외우는 것보다
- **DDD (도메인 주도 설계)**로 서비스를 설계할 때
- 무엇을 끝까지 따져야 하는지 구조화하고
- 그 결과로 **자기만의 `pg_*` 같은 설계 언어**를 만들 수 있는 사고 틀

이라고 보입니다.

그래서 아래처럼 정리하는 것이 가장 적절합니다.

---

## 목표 재정의

좋은 DDD (도메인 주도 설계) 설계자는
`Repository (리포지토리)`를 쓸지, `Active Record (액티브 레코드)`를 쓸지 먼저 고르지 않습니다.

먼저 다음을 정합니다.

1. **이 서비스에서 "존재한다"는 것이 무엇인가**
2. **무엇이 바뀌면 비즈니스적으로 중요한 변화인가**
3. **어디까지를 같은 것(one)으로 볼 것인가**
4. **어떤 규칙은 즉시 지켜져야 하고, 어떤 것은 나중에 맞춰도 되는가**
5. **기술적 경계가 아니라, 업무적 의미의 경계가 어디인가**

그 뒤에야 계층(layer), `Aggregate (집합체)`, `Repository (리포지토리)`, `Event (이벤트)`가 나옵니다.

---

## 깊이 있게 고려해야 하는 사유 축

### 1. 존재론: 이 서비스에서 "한 객체의 존재"는 무엇인가

가장 먼저 봐야 하는 것은 `table (테이블)`이 아니라 **존재의 정의**입니다.

| 질문                                      | 의미                                                           | 설계에 미치는 영향                           |
| ----------------------------------------- | -------------------------------------------------------------- | -------------------------------------------- |
| 이 객체는 언제 "생성"되나요?              | 입력 순간인지, 승인 순간인지, 저장 순간인지                    | 생성 `Event (이벤트)` 정의                   |
| 이 객체는 언제 "존재한다"고 볼 수 있나요? | 메모리상 생성인지, `DB (데이터베이스)` 반영인지, 외부 공개인지 | `Entity (엔티티)` / `Draft (초안)` 분리 여부 |
| 삭제는 소멸인가요, 비가시화인가요?        | `Hard Delete (하드 삭제)` / `Soft Delete (소프트 삭제)` 판단   | 생애주기 `Lifecycle (라이프사이클)` 모델링   |
| 식별자는 존재의 일부인가요?               | 저장 전에도 정체성이 있는가                                    | `Identity (식별자)` 생성 시점                |

이 축을 끝까지 밀어붙이면,
`Location`과 `PgLocation` 같은 구분이 나올 수도 있고,
반대로 하나의 `Entity (엔티티)`만으로 충분하다는 결론이 날 수도 있습니다.

즉, **패턴은 결과**이고,
그 전에 **존재 정의**가 있어야 합니다.

---

### 2. 시간론: 상태보다 중요한 것은 변화인가

DDD (도메인 주도 설계)에서 진짜 중요한 것은 종종 "현재 상태"가 아니라 **어떤 변화가 중요하냐**입니다.

| 질문                                                     | 의미                           | 설계 결과                                      |
| -------------------------------------------------------- | ------------------------------ | ---------------------------------------------- |
| 사용자는 현재 값보다 이력(history)을 더 중요하게 보나요? | 변경 추적 필요성               | `Versioning (버저닝)`, `Audit Log (감사 로그)` |
| 어떤 변화는 되돌릴 수 없나요?                            | irreversible action 여부       | append-only 구조 가능성                        |
| 변경의 단위는 필드(field)인가, 업무 행위(action)인가     | `update_status` vs `approve()` | `Anemic Model (빈약한 모델)` 여부 판단         |
| "언제" 바뀌었는지가 업무적으로 중요한가요?               | 시점 자체가 의미인지           | 이벤트 중심 설계 필요                          |

서비스가 단순 CRUD (생성/조회/수정/삭제)가 아니라면,
`state (상태)`보다 `transition (전이)`가 더 중요합니다.

이 관점이 생기면 질문이 바뀝니다.

```text
"어떤 컬럼(column)이 필요한가?"
→ "이 서비스에서 중요한 변화는 무엇인가?"
```

이 차이가 사유의 깊이를 만듭니다.

---

### 3. 정체성론: 무엇을 같은 것으로 볼 것인가

DDD (도메인 주도 설계)의 핵심은 사실 **same-ness (동일성)** 판단입니다.

| 질문                                               | 의미                          | 설계 영향                        |
| -------------------------------------------------- | ----------------------------- | -------------------------------- |
| 이 둘은 값이 같으면 같은 것인가요?                 | `Value Object (값 객체)` 여부 | 동등성 정의                      |
| 값이 달라도 같은 것인가요?                         | `Entity (엔티티)` 여부        | 식별자 중심 모델                 |
| 같은 이름이지만 버전이 다르면 같은 것인가요?       | 버전 독립성 판단              | `Versioned Entity (버전 엔티티)` |
| 외부 시스템에서 동일하다고 보는 기준은 무엇인가요? | 외부 참조 기준                | 외부 키(external key) 모델링     |

이걸 명확히 하지 않으면:

- `block`과 `block_version`
- `form_template`와 `template_version`
- `draft`와 `published`
- `user`와 `membership`

같은 구조가 전부 흐려집니다.

즉, 설계의 깊이는
**필드(field)를 잘 나누는 능력**이 아니라
**동일성(identity)을 잘 정의하는 능력**에 가깝습니다.

---

### 4. 경계론: 어디까지를 하나의 의미 단위로 볼 것인가

이게 `Aggregate (집합체)` 사고의 핵심입니다.

| 질문                                                             | 의미                         | 설계 결과                        |
| ---------------------------------------------------------------- | ---------------------------- | -------------------------------- |
| 어떤 규칙은 반드시 함께 지켜져야 하나요?                         | 강한 일관성 필요             | 같은 `Aggregate (집합체)` 가능성 |
| 무엇은 동시에 저장되지 않아도 되나요?                            | 약한 일관성 허용             | 분리된 `Aggregate (집합체)`      |
| 한 요청(request) 안에서 같이 바뀌지만, 사실 업무상 독립적인가요? | 기술적 동시성 vs 업무적 결합 | 분리 필요                        |
| 외부에서 참조할 때 내부를 다 알 필요가 있나요?                   | 캡슐화 범위                  | 루트(root) 설계                  |

핵심은 이것입니다.

```text
같이 조회된다고 같이 묶지 말고,
같이 깨지면 안 되는 규칙이 있는 것만 같이 묶어야 합니다.
```

즉 `Aggregate (집합체)`는
화면이나 `API (응용 프로그래밍 인터페이스)` 기준이 아니라,
**불변식(invariant) 기준**으로 잡아야 합니다.

---

### 5. 규칙론: 무엇이 불변식(invariant)인가

서비스 설계에서 가장 깊은 질문 중 하나입니다.

| 질문                                                | 의미                      | 설계 결과                                            |
| --------------------------------------------------- | ------------------------- | ---------------------------------------------------- |
| 절대로 깨지면 안 되는 규칙은 무엇인가요?            | core invariant            | `Aggregate (집합체)` 내부 규칙                       |
| 잠시 깨져도 나중에 복구되면 되는 규칙은 무엇인가요? | eventual consistency 허용 | 비동기 처리 가능                                     |
| 누가 이 규칙의 책임자(owner)인가요?                 | 책임 귀속                 | `Domain Service (도메인 서비스)` / `Entity (엔티티)` |
| 검증은 입력 검증인가, 도메인 규칙인가               | validation 층위           | `DTO (데이터 전송 객체)` vs domain 검증              |

예를 들면:

- 이메일 형식이 맞는가 → 입력 검증
- 이미 승인된 주문은 취소 불가 → 도메인 규칙
- 한 상담사는 같은 시간에 두 예약 불가 → 일관성 규칙
- 권한 버전이 다르면 재동기화 필요 → 시스템 규칙 + 도메인 규칙 사이

좋은 설계는 규칙을 많이 넣는 설계가 아니라,
**규칙의 종류를 분리하는 설계**입니다.

---

### 6. 언어론: 팀이 실제로 쓰는 말이 모델에 반영되는가

DDD (도메인 주도 설계)는 결국 `Ubiquitous Language (유비쿼터스 언어)`의 문제입니다.

| 질문                                               | 의미                                     | 설계 결과              |
| -------------------------------------------------- | ---------------------------------------- | ---------------------- |
| 현업은 "생성"이라고 하나요, "등록"이라고 하나요?   | 언어의 뉘앙스 차이                       | `create` vs `register` |
| "삭제"가 실제로는 "숨김"인가요?                    | 명칭과 실제 의미 차이                    | `delete` vs `archive`  |
| "프로젝트"가 진짜 프로젝트인가요, 문서 묶음인가요? | 이름과 개념의 불일치                     | 모델 재정의 필요       |
| 같은 단어를 부서별로 다르게 쓰나요?                | `Bounded Context (경계된 컨텍스트)` 신호 | 컨텍스트 분리          |

설계가 얕을 때는
모델 이름이 기술 용어를 따라갑니다.

설계가 깊어질수록
모델 이름이 **업무 언어를 정제한 결과**가 됩니다.

즉 `pg_*`도 중요한 것이 아니라,
왜 팀 언어상 그 접두사가 필요했는지 설명할 수 있어야 합니다.

---

### 7. 책임론: 누가 무엇을 결정해야 하는가

계층(layer)을 만드는 이유도 결국 책임 분리입니다.

| 책임 질문                              | 배치 후보                                    |
| -------------------------------------- | -------------------------------------------- |
| 이 규칙을 가장 잘 아는 것은 누구인가   | `Entity (엔티티)` / `Value Object (값 객체)` |
| 여러 `Aggregate (집합체)`를 조율하는가 | `Application Service (애플리케이션 서비스)`  |
| 저장 방법을 바꿔도 안 흔들려야 하는가  | `Repository (리포지토리)` 추상화             |
| 외부 시스템 호출인가                   | `Infrastructure (인프라)`                    |
| 사용자 입력 형식만 다루는가            | `Presentation (프레젠테이션)`                |

핵심 질문은 이것입니다.

```text
이 로직은 "무엇을 해야 하는지"를 아는가,
아니면 "어떻게 할지"를 아는가?
```

- 무엇을 해야 하는지 → domain 쪽
- 어떻게 할지 → infrastructure 쪽

이 구분이 흐려질수록 설계는 기술 중심이 되고,
선명할수록 서비스 중심이 됩니다.

---

## 사유를 결과로 만들기 위한 설계 프레임

아래 프레임으로 서비스마다 문서를 직접 써보시면 됩니다.

### 1. 서비스 존재 정의서

| 항목                               | 작성 내용                        |
| ---------------------------------- | -------------------------------- |
| 이 서비스의 핵심 객체는 무엇인가   | 예: 상담, 예약, 템플릿, 렌더     |
| 각 객체는 언제 존재한다고 보는가   | 입력/저장/승인/공개              |
| 각 객체의 생애주기는 어떻게 되는가 | draft → active → archived        |
| 식별자는 언제 부여되는가           | 생성 시 / 저장 시 / 외부 연동 시 |

---

### 2. 변화 정의서

| 항목                                      | 작성 내용                           |
| ----------------------------------------- | ----------------------------------- |
| 비즈니스적으로 중요한 변화는 무엇인가     | 승인됨, 배정됨, 게시됨              |
| 어떤 변화는 이벤트(event)로 남겨야 하는가 | `Published`, `Assigned`, `Canceled` |
| 이력(history)이 필요한가                  | 전체 / 일부 / 불필요                |
| 되돌릴 수 없는 변화는 무엇인가            | 제출, 결제, 전송 등                 |

---

### 3. 경계 정의서

| 항목                                | 작성 내용                      |
| ----------------------------------- | ------------------------------ |
| 즉시 일관성이 필요한 규칙           | 같은 `Aggregate (집합체)` 후보 |
| 나중에 맞춰도 되는 규칙             | 분리 가능                      |
| 어떤 데이터는 참조만 하면 되는가    | `id` 참조로 충분한가           |
| 어떤 것은 내부 구조를 몰라도 되는가 | 캡슐화 범위                    |

---

### 4. 규칙 정의서

| 항목           | 작성 내용                 |
| -------------- | ------------------------- |
| 입력 검증 규칙 | 형식, 범위, null 여부     |
| 도메인 규칙    | 업무상 불가/가능          |
| 정책 규칙      | 권한, 버전, 상태 전이     |
| 시스템 규칙    | 재시도, 중복 방지, 동기화 |

---

### 5. 언어 정의서

| 항목                  | 작성 내용                       |
| --------------------- | ------------------------------- |
| 현업이 쓰는 핵심 명사 | 상담, 사례, 회기, 초안          |
| 핵심 동사             | 등록, 배정, 승인, 종료          |
| 금지할 기술 용어      | row, record 같은 내부 용어      |
| 혼동되는 단어         | 삭제/숨김, 생성/등록, 게시/배포 |

---

## 이 수준의 사유를 만들기 위해 반복해야 하는 질문

아래 질문은 서비스마다 반복해서 던지셔야 합니다.

```text
1. 이 개념은 값(value)인가, 존재(entity)인가?
2. 이 변화는 데이터 수정인가, 업무 사건(event)인가?
3. 이 규칙은 지금 반드시 지켜야 하나, 나중에 맞춰도 되나?
4. 이 객체는 저장되어야 존재하나, 저장 전에도 존재하나?
5. 이 계층(layer)은 의미를 담는가, 기술을 담는가?
6. 이 이름은 개발자 언어인가, 업무 언어인가?
7. 이 묶음은 조회 편의 때문인가, 불변식(invariant) 때문인가?
```

이 질문에 끝까지 답하다 보면,
자연스럽게 다음이 나옵니다.

- 자기만의 `Aggregate (집합체)` 기준
- 자기만의 `Event (이벤트)` 기준
- 자기만의 `Repository (리포지토리)` 필요성 기준
- 자기만의 영속화(persistence) 철학
- 결국 자기만의 `pg_*` 같은 구조 언어

## 실전적으로는 어떻게 훈련하나

### 권장 훈련 순서

| 단계 | 훈련 내용                     | 산출물           |
| ---- | ----------------------------- | ---------------- |
| 1    | 화면 없이 서비스 설명 쓰기    | 순수 도메인 서술 |
| 2    | 명사/동사만 뽑기              | 언어 사전        |
| 3    | 중요한 변화만 추리기          | 이벤트 목록      |
| 4    | 절대 깨지면 안 되는 규칙 정리 | 불변식 목록      |
| 5    | 생애주기 정의                 | 상태 전이표      |
| 6    | 그 뒤에야 코드 구조 설계      | 계층/모듈 구조   |

즉, 먼저 코드를 짜는 것이 아니라:

```text
언어 → 존재 → 변화 → 규칙 → 경계 → 책임 → 코드
```

순으로 가셔야 합니다.

---

## 한 문장으로 요약하면

원하시는 수준의 사유는
**"어떤 패턴을 쓸까?"를 고민하는 수준이 아니라,
"이 서비스에서 존재, 변화, 규칙, 경계가 무엇인가?"를 끝까지 따져서
그 결과로 패턴이 나오게 하는 수준**입니다.
