# Worship Support Project

## 1. 프로젝트 개요

Worship Support Project는 청년부 예배 및 관련 사역에서 반복적으로 발생하는 수작업을 하나의 플랫폼 안에서 구조화하고, 재사용 가능하게 만들기 위한 시스템이다.  
이 프로젝트는 예배 준비 과정에서 필요한 각종 정보와 콘텐츠를 **블록(Block)** 단위로 관리하고, 이를 조합하여 **템플릿(Template)** 을 구성한 뒤, 최종적으로 다양한 형식으로 **공유·배포(Export/Publish)** 할 수 있도록 하는 것을 목표로 한다.

이 시스템이 해결하려는 문제는 다음과 같다.

- 예배 준비에 필요한 정보가 여러 사람, 여러 문서, 여러 채널에 흩어져 있음
- 주보, 찬양 목록, 프레젠테이션, 광고 카드뉴스 등이 각각 따로 제작됨
- 동일한 정보가 반복 입력되며, 수정 사항이 생기면 여러 산출물을 다시 수정해야 함
- 찬양 인도자, 예배지원팀, 찬양팀, 회중 등 대상별로 필요한 자료 형식이 다름

따라서 본 프로젝트는 예배/사역 콘텐츠를 데이터화하고, 조합 가능하게 만들며, 최종적으로 여러 출력물로 일관되게 배포하는 플랫폼을 지향한다.

---

## 2. 핵심 개념

### 2-1. Block

Block은 재사용 가능한 최소 콘텐츠 단위이다.  
예배 및 사역 운영에 필요한 정보 조각을 의미하며, 여러 템플릿에서 공통으로 활용될 수 있다.

현재 노션 기준으로 중요한 블록 예시는 다음과 같다.

- 주보에 필요한 순서 및 담당자 정보
- 찬양 목록 공지에 필요한 찬양 목록 정보
- 찬양팀 내부 공유용 음악 정보
- 텍스트 정보
- 이미지 정보
- 일정 정보
- 광고 정보
- 배경 정보
- 찬양 정보
- 찬양 목록 정보

즉 블록은 단순 텍스트 조각이 아니라, 예배 운영에 필요한 구조화된 데이터 단위로 이해해야 한다.

---

### 2-2. Template

Template은 블록들을 조합하여 특정 목적의 산출물을 만드는 틀이다.  
같은 블록이라도 어떤 템플릿에 들어가느냐에 따라 표현 방식과 배치가 달라질 수 있다.

현재 확인된 주요 템플릿 용례는 다음과 같다.

- 주보
- 예배 중 프레젠테이션
- 광고 카드뉴스
- 찬양팀 내부 공유용 음악 정보 문서

즉 템플릿은 “무엇을 만들 것인가”를 정의하는 그릇이며, 블록은 그 안에 담기는 재료라고 볼 수 있다.

---

### 2-3. Render

Render는 특정 템플릿 안에서 특정 블록을 어디에, 어떤 형태로 배치할지를 정의하는 영역이다.  
예를 들어 프레젠테이션 템플릿에서는 블록이 슬라이드 내 특정 위치와 크기로 배치될 수 있고, 카드뉴스에서는 정해진 레이아웃에 맞춰 배치될 수 있다.

따라서 Render는 실제 편집기(Editor)와 밀접하게 연결되는 개념이며, 화면 구성과 배치 규칙을 담당한다.

---

### 2-4. Export / Publish

완성된 템플릿은 최종적으로 공유가 필요한 대상에게 적절한 형태로 배포되어야 한다.  
현재 정리된 출력 형태는 다음과 같다.

- PDF
- Kakao 알림톡
- Slack message
- 배포된 Web link

즉 이 시스템은 단순 편집기에서 끝나는 것이 아니라, 제작된 결과물을 외부로 전달하는 배포 수단까지 포함하는 구조를 가진다.

## 3. 주요 유즈케이스

현재 노션에서 가장 중요한 유즈케이스 중 하나는 **“찬양 인도자를 지원한다”** 이다.

이 Task는 다음 흐름으로 이해할 수 있다.

1. 설교 본문을 묵상한다.
2. 찬양 목록을 구성한다.
3. 찬양 목록 공유를 위한 자료를 제작한다.
4. 필요한 대상에게 다양한 형식으로 공유한다.

여기서 2단계인 찬양 목록 구성은 다시 세부적으로 나뉜다.

- 곡 선정
- ref 확인
- key 확인
- sheet 확인
- song form 확인
- ment 정리
- tempo 정리
- 기타 필요한 정보 추가

그리고 3단계에서 만들어질 수 있는 자료는 다음과 같다.

- 악보 사진 자료 모음
- 텍스트 인사 및 설명
- 별도 PDF 문서
- YouTube playlist
- 회중 공유용 카드뉴스

이 유즈케이스는 본 프로젝트가 단순히 시각 자료를 만드는 툴이 아니라, **예배 준비 프로세스 전체를 지원하는 운영 도구**에 가깝다는 점을 보여준다.

---

## 4. 현재 이해한 시스템 흐름

현재까지의 정보를 기준으로 정리한 전체 흐름은 아래와 같다.

### Step 1. 블록 생성

예배 준비에 필요한 정보 조각을 Block으로 만든다.

### Step 2. 템플릿 구성

만들어진 Block들을 조합하여 목적에 맞는 Template을 구성한다.

### Step 3. 렌더링/편집

Template 안에서 Block의 위치, 크기, 순서, 표현 방식을 조정한다.

### Step 4. 출력 및 공유

완성된 결과물을 PDF, 메시지, 웹 링크 등으로 export한다.

이 흐름을 한 줄로 요약하면 다음과 같다.

> **정보를 구조화된 블록으로 저장하고, 조합하여 문서/화면으로 만들고, 다양한 채널로 배포하는 시스템**

## 5. 데이터 구조에 대한 현재 기준

기존 문서(project.txt, domain.txt)에서는 시스템 구조를 `auth / project / template / block` 중심으로 나누고 있었고, 블록 버전 관리, 템플릿-블록 조합, 렌더 배치 등을 상세히 다루고 있었다.  
다만 현재는 **노션에 있는 내용이 더 정확하고 최신 기준**이므로, 아래와 같은 원칙으로 이해해야 한다.

### 반영 원칙

- 노션 내용이 최신 기준이다.
- 기존 문서와 노션 내용이 겹치거나 충돌할 경우 노션 기준을 우선한다.
- 기존 문서는 초안/참고용 구조 문서로 간주한다.

### 현재까지 유지 가능한 핵심 구조

- Block 중심 구조
- Template 중심 산출물 구성
- Render를 통한 배치 개념
- Export/Publish 단계 포함
- 예배 운영 프로세스를 데이터화하려는 방향성

### 아직 정리가 더 필요한 부분

- 최종 기술 스택 확정
- 권한 구조
- 관리자 페이지 범위
- 편집기 진입 전 페이지 구조
- 이벤트, 로그, 예외 처리 기준
- DB 정책 및 트랜잭션 기준
- 실제 서비스 범위(예배지원 전체 vs 찬양 지원 중심)

## 6. 기술 스택 관련 현재 상태

기존 문서에는 다음과 같은 서버리스 중심 구성이 제시되어 있었다.

- Next.js (App Router)
- TypeScript
- Vercel
- Supabase
- Supabase Auth / Postgres / Storage / RLS

하지만 노션에는 다음과 같은 별도 내용도 존재한다.

- back: Python - FastAPI
- application: Flutter

즉 현재 기술 스택은 팀 내부에서 완전히 단일화된 상태로 보기 어렵고, 일부 구상은 서로 다르게 남아 있는 상황이다.

따라서 현재 시점에서는 기술 스택을 확정된 사실로 보기보다, 아래처럼 정리하는 것이 적절하다.

### 현재 판단

- 도메인 구조와 유즈케이스는 비교적 명확해지고 있음
- 그러나 구현 프레임워크와 런타임은 아직 팀 합의가 필요함
- 아키텍처 논의 시 기술 스택 자체보다 먼저 **도메인 구조, 책임 분리, 데이터 흐름**을 정리하는 것이 중요함

## 8. 아키텍처 정립 시 정해야 하는 항목

노션에 적힌 아키텍처 정립 항목은 다음과 같다.

- domain
- mvc
- exception
- log
- timestamp
- db 정책
- event
- transaction

이는 단순한 코드 스타일 결정이 아니라, 시스템을 어떻게 나눌지와 운영 기준을 어떻게 세울지에 관한 항목들이다.

각 항목의 의미를 현재 프로젝트 문맥에 맞춰 정리하면 다음과 같다.

### domain

서비스를 어떤 책임 단위로 나눌지 정하는 것  
예: auth, user, admin, project, template, render, block, song-list

### mvc

화면/입력/로직/데이터 흐름을 어떻게 분리할지 정하는 것  
단, 프레임워크에 따라 MVC를 그대로 쓰기보다 feature-based structure나 domain-based structure가 더 적절할 수 있음

### exception

어떤 오류를 어떻게 정의하고 응답할지 정하는 것  
예: 인증 오류, 권한 오류, 존재하지 않는 리소스, 잘못된 요청, 상태 충돌

### log

어떤 행동을 로그로 남길지 정하는 것  
예: 로그인, 프로젝트 생성, 템플릿 수정, 블록 변경, export 실행

### timestamp

생성/수정/삭제/배포 시간 등 시간을 어떤 기준으로 기록할지 정하는 것

### db 정책

삭제 정책, soft delete 여부, 관계 무결성, 프로젝트 간 데이터 섞임 방지 등 DB 규칙을 정하는 것

### event

특정 행동 이후 발생하는 후속 동작을 정의하는 것  
예: 템플릿 export 완료, 블록 업데이트 후 현재 버전 갱신, 공유 링크 생성

### transaction

한 번에 묶여서 반드시 같이 성공하거나 같이 실패해야 하는 작업을 정하는 것  
예: 블록 버전 생성 + current 지정, 찬양 리스트 재정렬, 템플릿 배치 수정

## 10. 성진 담당 범위 정리

성진의 담당 범위는 “편집기 내부 구현”보다는 **편집기로 들어가기 전까지의 서비스 구조**와 더 가깝다.

즉 성진이 우선 고민해야 하는 영역은 다음과 같다.

- 사용자는 어떻게 로그인하고 진입하는가
- 관리자 페이지는 어떤 목적과 권한을 가지는가
- 프로젝트는 어떻게 생성/조회/수정되는가
- 템플릿은 어떻게 생성되고 선택되는가
- 렌더는 어떤 단위에서 관리되는가
- 사용자가 편집기로 들어가기 전, 어떤 목록/상세/관리 화면이 필요한가

정리하면 성진의 아키텍처는 “에디터 자체”보다 아래 영역에 집중해야 한다.

- 서비스 진입 구조
- 사용자 흐름
- 관리자 흐름
- 프로젝트/템플릿 관리 구조
- 렌더 편집 전후 연결 구조

## 11. 기존 구조와 현재 구조의 관계

기존 문서 기준 구조는 비교적 기술 설계 중심이었다.

- auth / project / template / block 도메인 구분
- 버전 관리 중심 block 구조
- template + render 조합
- soft delete, audit log, RLS 등 운영 규칙

반면 노션 기준 구조는 실제 사용 흐름과 팀 작업 분담에 더 가까운 정보이다.

- 어떤 블록이 필요한지
- 어떤 템플릿이 필요한지
- 누구를 지원하는 시스템인지
- 어떤 결과물로 공유해야 하는지
- 누가 어떤 영역을 맡는지

즉 현재는 기존 설계 문서를 완전히 버리는 것이 아니라, 다음처럼 이해하는 것이 가장 적절하다.

### 정리

- 기존 문서: 기술 구조 초안
- 노션 문서: 실제 기획 및 최신 방향
- 앞으로의 아키텍처: 노션의 업무 흐름 위에 기존 기술 구조를 재정렬하는 작업

## 12. 현재 시점의 프로젝트 정의

현재까지의 내용을 종합하면 이 프로젝트는 다음과 같이 정의할 수 있다.

> Worship Support Project는 청년부 예배 준비에 필요한 다양한 콘텐츠와 정보를 블록 단위로 구조화하고, 이를 템플릿으로 조합하여, 예배지원팀·찬양팀·회중 등 각 대상에게 적절한 형식으로 배포할 수 있도록 하는 예배 운영 지원 플랫폼이다.

## 13. 현재 남아 있는 핵심 질문

아키텍처를 정립하기 전에 팀 차원에서 추가 합의가 필요한 질문은 다음과 같다.

1. 최종 기술 스택은 무엇인가?
2. 관리자 페이지는 누가 어떤 기능까지 사용하는가?
3. Project는 예배 단위인가, 시즌 단위인가, 팀 단위인가?
4. Template은 고정된 산출물인가, 자유 생성 가능한가?
5. Render는 단순 배치 정보인가, 편집 세션까지 포함하는가?
6. Block 버전 관리는 실제로 필요한가?
7. Export는 어떤 우선순위로 지원할 것인가?
8. Web link는 공개 링크인가, 로그인 기반 내부 링크인가?
9. 찬양 관련 데이터는 외부 자료와 연동할 것인가?
10. 1차 MVP 범위는 어디까지인가?

이 질문들에 대한 합의가 있어야 팀원별 아키텍처를 연결할 수 있다.

## 14. 제안하는 공통 이해 문장

> 이 프로젝트는 예배 준비 과정에서 필요한 정보와 콘텐츠를 블록 단위로 관리하고, 이를 템플릿으로 조합하여, 예배 중 사용 자료 및 공유용 결과물을 일관되게 제작·배포하기 위한 플랫폼이다. 현재 최신 기준은 노션 문서이며, 기존 설계 문서는 이를 기술적으로 구체화하기 위한 참고 초안으로 사용한다.
